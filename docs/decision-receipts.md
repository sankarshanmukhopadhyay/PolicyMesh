# Policy decision receipts

PolicyMesh v0.17.0 introduces machine-verifiable **policy decision receipts**. These receipts turn policy pull and admission decisions into durable evidence artifacts instead of transient CLI output.

## Why this matters

A federation operator needs more than the final policy hash. They need evidence of:

- which policy update was selected
- whether manifest trust checks passed
- whether quorum metadata was present
- whether the update was ready to apply, deferred, or rejected
- which reconciliation report and parent-chain recovery data supported the decision

That evidence is now written under `artifacts/receipts/<village_id>/`.

## Receipt shape

The JSON Schema lives at [`schemas/policy_decision_receipt.schema.json`](../schemas/policy_decision_receipt.schema.json).

Key fields:

- `decision`: `apply`, `defer`, or `reject`
- `reason_codes`: machine-readable decision reasons such as `ready_for_apply`, `awaiting_activation_time`, or `manifest_untrusted`
- `evidence.reconciliation_report_path`: pointer to the durable reconciliation artifact
- `evidence.quorum_summary`: the quorum model supplied by the update artifact

## Generated by policy pull

```bash
links policy pull https://peer.example.org ops --apply
```

On every pull, PolicyMesh now writes:

- `artifacts/reconciliation/<village_id>/pull.<timestamp>.json`
- `artifacts/receipts/<village_id>/policy_pull.<timestamp>.<policy_hash>.json`

If `LINKS_NODE_SIGNING_KEY_B64` is configured, the receipt is signed as an Ed25519 artifact.

## Verification

```bash
links policy verify-receipt artifacts/receipts/ops/policy_pull.20260417T073000Z.9b4a097db5fd.json
```

This checks both:

- receipt integrity via `receipt_id`
- receipt signature when signing material is present
