# v0.13.0 — Operator Adoption and Durable Policy Operations

This release packages four roadmap phases into one adoption-first increment. The focus is straightforward: make Links easier to trial, easier to recover, and easier to operate without turning it into a cathedral of governance ceremony.

## Highlights

### Reconciliation hardening
- `links policy pull` now writes a durable reconciliation artifact during pull flows.
- Selection is no longer a naive “latest wins” posture. Pulls now prefer a consistent remote head only when lineage is intact, and otherwise retain the local head as a deterministic fallback.
- Reconciliation reports now include `selected_source`, `selection_reason`, and `lineage_issues` for operator review.

### Storage abstraction with optional SQLite
- Added a pluggable storage layer with an optional SQLite backend controlled by `LINKS_STORAGE_BACKEND=sqlite`.
- Claim index rows, audit events, transparency entries, and policy state can now be mirrored into SQLite for more durable single-node and small-team pilots.
- Policy application records current state and policy history in transactional SQLite writes.

### Drift and transparency automation
- Added transparency checkpoint generation through `links drift checkpoint` and `scripts/transparency_checkpoint.py`.
- Checkpoints are written under `artifacts/transparency/<village_id>/...` for cross-node comparison and incident handling.
- Pull flows now persist signed or unsigned manifest artifacts alongside the selected policy artifact.

### Production deployment guidance
- Added `docs/deploy/sqlite-backend.md` for the durable local backend posture.
- Added `docs/deploy/operator-playbook.md` with concrete response guidance for drift, forks, lineage gaps, checkpoint mismatches, and signer incidents.
- Updated README and roadmap language to reflect shipped capabilities versus future work.

## Upgrade notes
- Filesystem artifacts remain the default operational surface. SQLite is optional and intended for operators who want stronger local durability without introducing a separate database service.
- Existing integrations keep working without storage configuration changes.
- For pilot deployments, preserve both the SQLite file and filesystem artifacts during backup.
